# 한림병원 직원용 포탈 프로젝트 규칙 (Project Rules) v3.0

## 1. 기술 스택 및 아키텍처 (Tech Stack & Architecture)
- **Framework**: 빠른 렌더링과 API 서버 통합 구축을 위해 **Next.js (React 기반 풀스택 프레임워크)**를 사용한다.
- **Frontend**: React 컴포넌트 기반으로 개발하며, **Tailwind CSS**를 사용하여 빠르고 일관된 스타일링을 수행한다. 복잡한 전역 상태(현재 부서, 로그인 정보 등)는 `Zustand`와 같은 경량 상태 관리 라이브러리를 활용한다.
- **Backend (API)**: 별도의 서버 배포 없이 Next.js의 **Route Handlers (API Routes)**를 백엔드로 사용한다. 프론트엔드는 직접 DB와 통신하지 않고 이 API를 호출한다.
- **DB & 데이터 페칭 (핵심)**: Google Spreadsheet를 DB로 사용한다(`google-spreadsheet` 패키지 활용). 구글 시트의 느린 응답 속도를 극복하기 위해 다음 전략을 필수적으로 적용한다:
  1. **캐싱(Caching)**: 데이터 조회 시 Next.js의 캐시 기능(ISR 등)을 활용하여 구글 시트 직접 접근을 최소화하고 응답 속도를 높인다.
  2. **낙관적 업데이트 (Optimistic UI)**: 사용자가 데이터를 수정(예: 서명, 장비 확인)할 때, DB 응답을 기다리지 않고 프론트엔드 UI를 먼저 변경한 뒤 백그라운드에서 구글 시트를 업데이트하여 앱이 즉각적으로 반응하는 것처럼 느끼게 한다.

## 2. 디자인 시스템 (Toss 스타일 지향)
- **UI/UX 방향성**: 토스(Toss) 앱처럼 직관적이고 여백이 넉넉하며, 부드러운 곡선과 은은한 그림자를 활용한 미니멀하고 모던한 룩을 지향한다.
- **배경 및 카드**: 전체 배경은 연한 쿨그레이(`#f2f4f6`), 콘텐츠 카드는 순백색(`#ffffff`)을 사용하여 영역을 부드럽게 분리한다.
- **컴포넌트 스타일**: 모든 버튼과 입력창, 카드는 둥근 모서리(`rounded-xl`, `rounded-2xl`)를 적용하고, 토스 특유의 옅은 테두리와 포커스 시 부드러운 전환 효과(`transition-all`)를 준다.
- **색상 및 시인성**: 병원 업무 환경을 고려해 메인 포인트 컬러는 메디컬 블루(`#004b8d`) 계열을 사용하고, 경고/알림은 오렌지(`#ff7a00`) 계열을 사용하여 시각적 피로도를 낮추고 가독성을 극대화한다.

## 3. 반응형 레이아웃 규칙 (Responsive Design)
- **컴포넌트 분리**: 화면 레이아웃은 재사용 가능한 React 컴포넌트(예: `<Sidebar />`, `<BottomNav />`, `<Header />`)로 명확히 분리하여 관리한다.
- **PC 환경 (md 이상)**: 화면 좌측에 고정된 사이드바(Left Sidebar) 메뉴 구조를 적용하여 넓은 화면을 효율적으로 사용한다.
- **모바일 환경 (md 미만)**: 왼쪽 사이드바를 숨기고, 화면 하단에 고정된 네비게이션 바(Bottom Navigation Bar)를 탭 형식으로 배치하여 한 손 조작 편의성을 높인다.
- **모바일 방어적 CSS**: 좁은 모바일 화면에서 UI가 가로로 삐져나가거나 화면이 축소/확대되는 것을 막기 위해 `flex-wrap`, `min-w-0`, `w-full`, `truncate` 처리를 꼼꼼히 점검하여 방어적인 CSS를 작성한다.
- **테이블 레이아웃 최적화 (필수 유지)**: 다수의 열(예: 31일치 날짜)을 포함하는 테이블(근무표 등)은 다음 두 가지 원칙을 반드시 유지한다.
  1. **가로 너비 최적화**: PC 화면에서 스크롤 없이 모든 열이 한 화면에 꽉 차게 보이도록 `table-fixed w-full min-w-[...px]` 구조를 적용하고, 각 날짜 셀은 고정 너비(`w-[...]`) 대신 최소 너비(`min-w-[...]`)를 사용하여 화면 크기에 맞춰 유연하게 줄어들도록 설계한다.
  2. **세로 높이 꽉 차게 확장**: 화면 하단의 빈 공간이 남지 않도록 테이블 엘리먼트에 `h-full` (또는 `min-h-full`) 클래스를 적용하여, 각 행(row)의 높이가 남은 공간에 맞게 꽉 차게 늘어나도록 구성한다.

## 4. 유연성 및 부서 확장성 (Scalability)
- **부서별 확장 구조**: 현재는 '기능검사실' 타겟이지만, 추후 병동, 외래, 수술실 등 타 부서가 추가될 수 있도록 시스템을 설계한다.
- **데이터 스키마**: 데이터베이스(구글 시트) 설계 및 프론트엔드 타입(TypeScript 인터페이스 등) 정의 시, 반드시 `department`(부서) 혹은 `ward`(병동) 속성을 포함시킨다.
- **상태 관리**: 로그인한 사용자의 소속 부서를 전역 상태로 관리하여, 해당 부서의 데이터만 렌더링되도록 동적 라우팅 및 필터링 로직을 구성한다.

## 5. AI 코드 작성 주의사항 (Strict Constraints)
- **완전한 컴포넌트/파일 제공**: 코드를 수정하거나 기능을 추가할 때 중간에 코드를 생략(`// 기존 코드 동일` 등)하지 말고, 항상 즉시 복사-붙여넣기하여 실행 가능한 완전한 파일(또는 컴포넌트) 코드를 제공한다.
- **관심사 분리 (SoC)**: API 통신 로직, 상태 관리 로직, UI 렌더링 로직을 한 파일에 섞지 말고, React의 커스텀 훅(Custom Hook)이나 별도의 유틸리티 함수로 깔끔하게 분리하여 제안한다.
